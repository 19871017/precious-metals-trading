# 交易系统重构 - 第一阶段需求规格

## 背景

当前贵金属交易平台核心交易链路存在稳定性问题,需要进行架构重构以确保订单、持仓、资金三大模块具备生产级一致性。

## 需求概述

### 一、订单系统

**需求 1.1**: 引入订单处理队列机制
- 使用 BullMQ + Redis 实现订单处理队列
- 所有下单请求必须进入队列处理
- 禁止同步直接撮合订单

**需求 1.2**: 实现订单状态机
- 定义订单状态转换: CREATED → PROCESSING → FILLED → CLOSED → FAILED
- 确保订单状态转换的原子性和一致性
- 防止订单状态的非法转换

**需求 1.3**: 防止重复订单执行
- 使用分布式锁确保同一订单不会被多次处理
- 实现幂等性机制,重复请求应返回相同结果

### 二、持仓系统

**需求 2.1**: 平仓逻辑增加分布式锁
- 使用 Redis 分布式锁保护平仓操作
- 防止并发平仓导致的数据不一致

**需求 2.2**: 防止重复平仓
- 检查持仓状态,避免对已平仓的持仓进行操作
- 使用唯一约束或状态检查确保持仓只被平仓一次

**需求 2.3**: 平仓失败事务回滚
- 平仓操作必须在数据库事务中执行
- 平仓失败时必须回滚所有相关变更
- 确保保证金释放、盈亏计算等操作的原子性

**需求 2.4**: 盈亏计算解耦
- 盈亏计算不应依赖实时行情数据
- 使用历史价格快照进行盈亏计算
- 确保盈亏计算的确定性

### 三、资金系统

**需求 3.1**: 资金变动事务化
- 所有资金变动操作必须在数据库事务中执行
- 包括充值、提现、手续费、盈亏等所有资金操作
- 确保资金变动的原子性和一致性

**需求 3.2**: 增加资金操作审计日志
- 记录所有资金变动的审计日志
- 包括操作类型、金额、操作前余额、操作后余额、关联订单、时间戳等
- 审计日志独立存储,不可篡改

**需求 3.3**: 防止余额并发修改
- 使用数据库行级锁或乐观锁保护账户余额
- 使用分布式锁防止并发修改
- 确保余额计算的准确性

**需求 3.4**: 增加账户冻结字段
- 在账户表中增加 frozen_amount 字段
- 支持冻结部分余额用于保证金或风控
- 实现冻结和解冻操作的原子性

### 四、系统约束

**需求 4.1**: 保持 API 兼容性
- 不修改现有业务逻辑接口
- 保持现有 API 响应格式不变
- 重构对外部调用方透明

**需求 4.2**: 可追溯性
- 输出修改架构说明文档
- 输出风险修复报告
- 记录所有关键操作的日志

## 验收标准

### 订单系统
- [ ] 所有下单请求通过 BullMQ 队列处理
- [ ] 订单状态机完整实现且状态转换正确
- [ ] 重复订单请求返回相同结果
- [ ] 订单处理失败能正确回滚到 FAILED 状态

### 持仓系统
- [ ] 平仓操作使用分布式锁保护
- [ ] 已平仓的持仓无法再次平仓
- [ ] 平仓失败能正确回滚
- [ ] 盈亏计算使用快照数据,不依赖实时行情

### 资金系统
- [ ] 所有资金变动在事务中完成
- [ ] 审计日志完整记录所有资金操作
- [ ] 并发修改余额时不会出现数据不一致
- [ ] 账户冻结/解冻操作正确执行

### 系统约束
- [ ] 现有 API 接口保持兼容
- [ ] 架构说明文档完整
- [ ] 风险修复报告完整
- [ ] 关键操作日志完整记录

## 非功能性需求

### 性能要求
- 订单处理延迟不超过 100ms (不含网络延迟)
- 队列处理吞吐量不低于 1000 订单/秒
- 数据库事务执行时间不超过 50ms

### 可靠性要求
- 系统可用性不低于 99.9%
- 数据一致性达到强一致性
- 故障恢复时间不超过 5 分钟

### 安全性要求
- 所有资金操作必须审计
- 分布式锁超时时间合理设置
- 防止重放攻击
